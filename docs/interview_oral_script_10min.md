# Agent 后端平台 10 分钟口语化逐字稿

> 使用方式：按时间顺序背诵。  
> 目标：让你在 10 分钟内讲清“定位、架构、核心链路、难点、结果、演进”。  
> 建议：先读熟，再用自己的习惯词汇替换 20%-30%。

---

## 0:00 - 0:30 开场定位（30秒）

我这个项目是一个 Agent 后端执行平台。  
核心不是做聊天接口，而是把大模型能力升级为可执行任务系统。  
系统支持工具调用、计划执行、权限审批、流式输出、沙箱执行和会话持久化，重点解决 AI 应用“能答但不稳、能跑但不可控”的问题。

---

## 0:30 - 1:30 为什么做这个（1分钟）

我当时看到三个典型痛点：

1. 聊天式后端只能输出文本，不能可靠落地执行任务。
2. 有副作用操作时缺乏治理，比如写文件、执行命令，很容易误操作。
3. 执行过程不可审计，出问题后很难复盘。

所以我把目标定义为三件事：

1. 构建可控执行循环（Query Loop）
2. 建立治理机制（plan/execute + permission）
3. 做稳定契约（SSE 事件 + 持久化 + 错误语义统一）

---

## 1:30 - 3:00 架构怎么拆（1分30秒）

整体我是分四层：

1. API 接入层  
用 FastAPI 暴露 `/agent`、`/sandbox`、`/providers`、`/files` 等接口，统一协议和错误处理。

2. 编排层  
在 `agent_service` 里把请求转成执行任务，负责上下文裁剪、计划解析、权限回调、错误标准化。

3. SDK 执行层  
`Agent + QueryEngine` 负责模型交互和工具循环。模型每一轮返回文本和工具调用，工具执行结果再回喂模型。

4. 基础设施层  
包括 Provider 适配、Sandbox 管理与池化、SQLite 持久化和配置回退。

一句话概括：  
“上层保证协议稳定，中层保证执行可控，下层保证能力可扩展。”

---

## 3:00 - 5:00 核心链路（2分钟）

我讲一条最典型的 `/agent` 链路。

第一步，用户发起 `/agent`。  
API 层把请求交给编排层，编排层组装 AgentOptions：模型配置、工作目录、权限模式、skills、mcp 等参数。

第二步，进入 QueryEngine 循环。  
引擎调用 provider 获取模型输出。输出分两类：

1. `text`：直接输出给前端
2. `tool_calls`：进入工具执行

第三步，工具执行。  
工具执行后生成 `tool_result`，引擎把结果写回消息上下文，继续下一轮模型推理，直到没有工具调用再收敛为 `result`。

第四步，事件流和持久化。  
整个过程事件化输出为 SSE：`text/tool_use/tool_result/result/error/done`。  
如果带 `taskId`，我会把事件落到 SQLite 的 `messages/tasks/sessions` 里，保证可审计可回放。

---

## 5:00 - 6:30 风险治理（1分30秒）

我做了两层治理：

第一层：计划-执行双阶段。  
`/agent/plan` 先让模型给结构化计划，再由 `/agent/execute` 执行，避免直接进入高风险动作。

第二层：权限审批闭环。  
对非只读工具，QueryEngine 会发 `permission_request`。  
外部通过 `/agent/permission` 反馈 approve/deny。  
deny 时不会执行副作用操作，而是返回工具错误事件。

这套机制把“能执行”变成“可控地执行”。

---

## 6:30 - 7:30 技术难点与解决（1分钟）

我遇到三个典型难点：

1. 模型输出不稳定  
我在编排层做了鲁棒 JSON 解析，支持 code block、混合文本、脏 JSON 提取和 direct_answer fallback。

2. Provider 协议差异  
我抽象了 provider 接口，分别适配 Anthropic 和 OpenAI-compatible，统一上层调用语义。

3. 并发一致性  
我在 task 索引分配时用了事务原子预留，避免并发下 `task_index` 冲突。

---

## 7:30 - 8:30 可扩展性（1分钟）

扩展能力主要体现在两块：

1. Tool 系统  
我把工具统一成 `ToolDefinition`，目前有 36 个基础工具，新增工具只要按 schema 接口注册就能接入执行循环。

2. MCP 动态注入  
支持 stdio/http/sse 三种传输，运行时发现外部工具后自动注入为 `mcp__<server>__<tool>`，不用改主流程。

另外 sandbox 支持 provider fallback 和可选池化，兼顾可用性和启动性能。

---

## 8:30 - 9:10 结果与质量（40秒）

截至当前版本：

1. 56 个 API 路由
2. 36 个工具定义
3. 71 个自动化测试全部通过
4. 覆盖了契约、SSE 顺序、权限链路、MCP、sandbox 池化和回退等关键场景

所以这个项目不是 Demo，而是可持续迭代的后端执行底座。

---

## 9:10 - 10:00 收尾与演进（50秒）

后续我计划做三件事：

1. 加强可观测性：补齐 metrics 和 tracing，比如 tool failure rate、fallback rate、p95 latency。
2. 提升吞吐：引入异步队列和分布式执行器，避免单节点压力。
3. 强化多租户治理：tenant 级权限策略、配额和审计隔离。

我这个项目的核心价值在于：  
把 AI 能力从“文本回复”推进到“可治理、可审计、可扩展的执行系统”。

---

## 备忘：讲的时候的语气策略

1. 不要只报名词，要一直围绕“问题 -> 方案 -> 效果”。
2. 不要说“我什么都做了”，重点讲架构和关键判断。
3. 遇到深问，先给结论，再给实现细节。
4. 每段都能落到一个关键词：可控、稳定、扩展、审计。
